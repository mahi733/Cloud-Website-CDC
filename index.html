Unit I · Introduction
Introduction to Cloud Computing
Understand what cloud computing is, its historical roots, the three service models (IaaS, PaaS, SaaS), deployment types, and the pros and cons of moving to the cloud.
1.1 Defining Cloud Computing

Cloud computing is the delivery of computing services — including servers, storage, databases, networking, software, analytics, and intelligence — over the Internet ("the cloud") to offer faster innovation, flexible resources, and economies of scale.

The NIST definition describes five essential characteristics:

  • On-demand self-service — Users provision resources automatically without requiring human interaction with the service provider.
  • Broad network access — Capabilities available over the network and accessible through standard mechanisms (phones, tablets, laptops).
  • Resource pooling — Provider's resources pooled to serve multiple consumers using a multi-tenant model.
  • Rapid elasticity — Capabilities can be elastically provisioned and released to scale rapidly.
  • Measured service — Cloud systems automatically control and optimize resource use (pay-as-you-go).
1.2 Roots of Cloud Computing

Mainframes to Cloud: The journey began with large mainframe computers in the 1950s–60s shared via dumb terminals. Time-sharing on mainframes was the first "multi-tenant" computing model.

Grid Computing: Connects many computers across a network to solve large problems by sharing computation power — precursor to cloud's distributed model.

Utility Computing: Coined by John McCarthy (1961), the idea of computing as a public utility like electricity. Users pay only for what they consume.

Hardware Virtualization: The ability to run multiple virtual machines on a single physical machine — the key enabling technology that made cloud economically feasible.

1.4 Types of Cloud Deployment
  • Public Cloud: Infrastructure owned by a third-party provider and shared among many customers. Highly scalable, pay-as-you-go. (AWS, Azure, GCP)
  • Private Cloud: Infrastructure exclusively for one organization. Can be on-premises or hosted. More control and security, but higher cost.
  • Hybrid Cloud: Combination of public and private clouds, connected by technology allowing data and applications to move between them. Best of both worlds.
  • Community Cloud: Shared by several organizations with common concerns (same industry, compliance requirements).

Unit II · Core Concept
Virtualization & Hypervisors
Explore how virtualization abstracts physical hardware to run multiple isolated environments, and how hypervisors manage those virtual machines.
2.1 What is Virtualization?

Virtualization is the process of creating a software-based (virtual) representation of something — be it compute, storage, networking, or an entire operating system — rather than using an actual physical version.

Key characteristics: Partitioning (run multiple OSes on one machine), Isolation (faults in one VM don't affect others), Encapsulation (entire state saved in files), Hardware Independence (VMs run on any compatible hardware).

Key Insight Virtualization is the foundational technology of cloud computing. Without it, efficient multi-tenancy, elasticity, and resource pooling would be impossible at scale.
2.3 Types of Virtualization (by technique)
  • Full Virtualization: The VM completely simulates the underlying hardware. Guest OS needs no modification. Examples: VMware Workstation, VirtualBox. High compatibility but slight performance overhead.
  • Paravirtualization: Guest OS is modified to be aware it's running in a virtual environment and communicates directly with the hypervisor. Faster but requires OS modification. Example: Xen with modified kernel.
  • Partial Virtualization: Only some hardware resources are virtualized. Rarely used today.
  • OS-Level Virtualization (Containers): The OS itself provides isolation. Multiple containers share the same OS kernel. Very lightweight. Example: Docker, LXC.
Full Virtualization vs. Paravirtualization vs. Containers
2.4 Types of Virtualization (by resource)
  • Storage Virtualization: Aggregates physical storage from multiple devices into a single logical storage pool (e.g., SAN virtualization).
  • Network Virtualization: Combines hardware and software network resources into a single, software-based administrative entity (VLANs, SDN).
  • Desktop Virtualization: Runs desktop OS from a central server; users access via thin clients. Examples: Citrix, VMware Horizon, Chrome Remote Desktop.
  • Application Virtualization: Applications run in a virtual environment separate from the underlying OS (e.g., Java JVM, App-V).
2.5 Hypervisors & Virtual Machines

A hypervisor (Virtual Machine Monitor — VMM) is the software layer that creates and manages virtual machines by allocating physical resources.

Unit III · Infrastructure
Data Center Architecture
Learn the building blocks of a modern data center — physical infrastructure, networking topologies, Software-Defined Networking, and automation strategies.
3.1 Data Center Fundamentals

A data center is a facility used to house computing infrastructure — servers, storage, networking equipment — along with the power, cooling, and security systems needed to keep them running continuously.

Key Physical Components:

  • Servers (Compute): Rack-mount servers, blade servers, modular compute units.
  • Storage Systems: SAN (Storage Area Networks), NAS (Network Attached Storage), object storage arrays.
  • Networking: Switches, routers, load balancers, firewalls, cables.
  • Power Infrastructure: UPS (Uninterruptible Power Supply), PDU (Power Distribution Units), diesel generators for redundancy.
  • Cooling: CRAC units (Computer Room Air Conditioning), hot/cold aisle containment, liquid cooling for high-density racks.
  • Physical Security: Biometric access, CCTV, cages, mantraps.
Tier Ratings Data centers are classified in Tiers I–IV by the Uptime Institute. Tier IV provides 99.9999% uptime with fully redundant power and cooling paths.
3.2 Data Center Network Topologies

The internal network of a data center determines performance, fault tolerance, and scalability.

  • Three-Tier (Core-Distribution-Access): Classic hierarchical model. Core layer handles high-speed backbone; distribution layer manages routing between subnets; access layer connects servers. Simple to manage but limited east-west traffic performance.
  • Leaf-Spine (Fat-Tree): Modern flat two-layer design used in hyperscale data centers. Every leaf (access) switch connects to every spine switch. Provides equal-cost paths and high bandwidth for east-west (server-to-server) traffic. Used by Google, Facebook, AWS.
  • Mesh Topology: Every device connects to every other. Redundant but expensive — used for very critical nodes.
Leaf-Spine Architecture
3.2.2 SDN — Software-Defined Networking

SDN separates the network's control plane (deciding where traffic goes) from the data plane (moving traffic). A centralized SDN controller programs network behavior across the whole infrastructure via open APIs (e.g., OpenFlow).

Benefits in Data Centers: Centralized visibility, dynamic traffic engineering, faster provisioning, vendor independence, and the ability to automate responses to load or failures.

Example tools: OpenFlow protocol, OpenDaylight controller, VMware NSX, Cisco ACI.

Mininet is an open-source emulator for SDN networks. It lets you build a virtual network with hundreds of nodes on a single laptop — ideal for lab exercises (PrO 7 in your syllabus).
3.3 Automation & Infrastructure as Code (IaC)

Data Center Automation uses scripts and tools to provision, configure, and manage infrastructure with minimal manual intervention.

Infrastructure as Code (IaC) treats infrastructure definitions (servers, networks, firewalls) as version-controlled code files, making deployments repeatable and consistent.

  • Terraform: Declarative IaC tool by HashiCorp. Define desired state; Terraform figures out what to create/destroy.
  • Ansible: Agentless configuration management using YAML playbooks.
  • Chef / Puppet: Configuration management with recipes/manifests.

Scalability & Elasticity: Cloud data centers scale horizontally (add more machines) rather than vertically (bigger machine). Auto-scaling groups automatically add or remove compute nodes based on real-time demand metrics.

Unit IV · Data Management
Cloud Storage & Database Services
Understand the three storage paradigms, consistency models, and the SQL vs NoSQL database landscape in cloud environments.
4.1 Cloud Storage Solutions

Cloud storage comes in three fundamental forms, each optimized for different workloads:

Three Cloud Storage Paradigms

Data Consistency Models:

  • Strong Consistency: Every read gets the most recent write. Safer but slower (traditional RDBMS).
  • Eventual Consistency: Updates propagate over time; temporarily stale reads are possible. Faster and more scalable (DynamoDB, Cassandra default).
  • Durability: Ensured through replication (typically 3 copies across different fault domains).

Data Scaling Strategies:

  • Replication: Copies of the database kept in sync. A Primary handles writes; Replicas handle reads. Improves read performance and availability.
  • Sharding (Horizontal Partitioning): Data split across multiple database servers (shards) based on a shard key. Each shard handles a portion of the data independently — allows limitless horizontal scaling.

Unit V · Security
Cloud Security & Compliance
Explore the shared responsibility model, identity and access management, data encryption strategies, SLA metrics, and the DevSecOps philosophy.
5.1 Security Challenges in the Cloud

Moving to the cloud introduces new security considerations. The Shared Responsibility Model defines which security tasks belong to the cloud provider and which belong to the customer.

  • Data breaches — Misconfigured S3 buckets, leaked credentials, injection attacks.
  • Insecure APIs — Publicly exposed management APIs without proper authentication.
  • Account hijacking — Phishing or credential stuffing attacks on cloud console accounts.
  • Insider threats — Privileged users abusing access.
  • Compliance risks — Storing regulated data (PII, PCI-DSS, HIPAA) in the wrong region or without proper encryption.
Shared Responsibility In IaaS: Provider secures the physical data center and hypervisor. YOU secure the OS, middleware, applications, and data. In SaaS: Provider secures almost everything; you manage your data and user access.
5.1.2 Identity & Access Management (IAM)

IAM is the framework of policies and technologies for ensuring that the right people and systems have access to the right resources, at the right time, for the right reasons.

  • Authentication: Verifying who you are (passwords, MFA, certificates, biometrics).
  • Authorization: What you're allowed to do (RBAC, ABAC, policies).
  • Role-Based Access Control (RBAC): Permissions assigned to roles (Admin, Developer, ReadOnly), not individual users.
  • Principle of Least Privilege: Every entity gets only the minimum permissions needed to perform its function. Critical for reducing attack surface.
  • Multi-Factor Authentication (MFA): Requires two or more verification factors (password + OTP + biometric).
IAM Flow: Authentication → Authorization → Resource Access
5.2 Data Security Technologies
  • Encryption at Rest: Data stored on disk is encrypted (AES-256). Even if storage media is stolen, data is unreadable. Provided by default in most cloud services.
  • Encryption in Transit: Data moving over the network is encrypted using TLS/SSL. Protects against man-in-the-middle attacks.
  • Key Management Service (KMS): Centralized service to create, manage, and rotate encryption keys (AWS KMS, Google Cloud KMS, Azure Key Vault).
  • Data Masking & Tokenization: Substitute sensitive data with non-sensitive equivalents for non-production environments.
  • VPC (Virtual Private Cloud): Logically isolated network within the cloud. Fine-grained control over subnets, routing, and network ACLs.
5.3 SLAs & DevSecOps

Service Level Agreement (SLA): A contract between a cloud provider and customer defining performance guarantees.

  • Availability (Uptime): E.g., 99.99% = ~52 minutes downtime/year. "Five nines" (99.999%) = ~5 minutes/year.
  • RTO (Recovery Time Objective): Maximum acceptable time to restore service after failure.
  • RPO (Recovery Point Objective): Maximum acceptable data loss measured in time (how old can a backup be?).

DevSecOps integrates security practices into the DevOps pipeline — "shift left" means testing for security vulnerabilities early in the development lifecycle, not just before release. Tools: SAST (static code analysis), DAST (dynamic testing), container image scanning.

Unit VI · Future Technologies
Emerging Technologies with Cloud
Survey the frontier: serverless computing, edge and fog computing, AI/ML services, blockchain on cloud, 5G networking, Kubernetes, and containers.
6.1 Serverless Computing

In serverless computing, developers write code (functions) without provisioning or managing servers. The cloud platform automatically scales the code from zero to global scale and you pay only per execution.

  • AWS Lambda, Google Cloud Functions, Azure Functions — Run code in response to events (HTTP request, file upload, database change).
  • Benefits: No server management, automatic scaling, event-driven, extremely cost-efficient for sporadic workloads.
  • Limitations: Cold starts (latency on first invocation), execution time limits, stateless by design.
Function as a Service (FaaS) is the core of serverless. The unit of deployment is a single function, not an app or container.
6.2 Edge & Fog Computing

Both paradigms move computation closer to where data is generated, reducing latency and bandwidth usage.

Compute Hierarchy: Cloud → Fog → Edge → Devices

Edge Computing places compute directly at or near the data source (IoT sensors, cameras, vehicles). Millisecond response times. Critical for autonomous vehicles, AR/VR, industrial automation.

Fog Computing (by Cisco) adds an intermediate layer between edge devices and the cloud. Fog nodes are more powerful than edge nodes and can aggregate, filter, and pre-process data before sending to cloud.

6.3 AI & Machine Learning with Cloud

Cloud providers offer AI/ML services at multiple abstraction levels:

  • Pre-trained APIs: Vision AI, Speech-to-Text, Translation — use without any ML knowledge (AWS Rekognition, Google Vision API).
  • ML Platforms: Build, train, and deploy custom models (AWS SageMaker, Google Vertex AI, Azure ML).
  • GPU/TPU Clusters: Cloud-based accelerators for training large models (A100 clusters, Google TPU Pods).

Cloud democratizes AI by eliminating the need to buy expensive hardware for model training.

6.4 DLT (Blockchain) with Cloud

Distributed Ledger Technology (DLT), including blockchain, maintains synchronized records across multiple nodes without a central authority. On cloud, managed blockchain services eliminate the need to set up nodes manually.

  • Amazon Managed Blockchain (supports Hyperledger Fabric, Ethereum)
  • Azure Blockchain Service

Use cases: supply chain transparency, smart contracts, digital identity, decentralized finance (DeFi).

6.6 Kubernetes & Containers

Docker packages applications and their dependencies into portable containers. A container is like a lightweight VM — it includes the app + runtime + libraries but shares the host OS kernel.

Kubernetes (K8s) is an open-source container orchestration system that automates deploying, scaling, and managing containerized applications.

  • Pod: Smallest deployable unit; one or more containers sharing network and storage.
  • Node: Physical or virtual machine running pods.
  • Cluster: A set of nodes managed by a control plane.
  • Service: Stable network endpoint for a set of pods (load balancing).
  • Deployment: Desired state declaration — K8s ensures actual state matches.
Kubernetes Cluster Architecture
6.5 5G & Cloud-Native Networking

5G provides ultra-low latency (~1ms), massive device density, and high bandwidth — enabling new cloud applications that were previously impractical.

  • Network Slicing: 5G allows virtual dedicated networks on shared physical infrastructure — each slice with its own QoS parameters.
  • Mobile Edge Computing (MEC): Placing cloud compute at the 5G base station edge to serve mobile users with near-zero latency.
  • Cloud-Native Network Functions (CNFs): Telecom functions (firewall, load balancer) deployed as containers/microservices rather than dedicated hardware boxes.
